Create a new design system from scratch. `$ARGUMENTS` may name the product or brand.

**Load `${CLAUDE_PLUGIN_ROOT}/skills/design-system-conventions/SKILL.md` first** ‚Äî it is the definition of good you build to (three conventions, two-tier token spine + parity, figc rules, the universal plan‚Üíapprove‚Üíexecute contract). Read it, don't reconstruct it from memory.

This command is **plan ‚Üí approve ‚Üí execute**. The plan phase writes nothing to Figma, code, tokens, or the manifest.

## Plan phase (read-only)

1. **Brand interview ‚Äî run in this conversation.** You conduct it directly with AskUserQuestion / batched questions; do NOT dispatch it as a subagent (a subagent cannot interview the human). Follow the `@design-system-wizard:brand-interviewer` question set: existing product or greenfield; brand / main color; typography (font choices, tone of type); overall tone and personality; density & corner style; any hard brand constraints. **Ask the theming-modes question explicitly and always** ‚Äî Light only ¬∑ Light + Dark (default) ¬∑ Dark only ‚Äî as its own choice; it sets the theming axis, decides whether `semantic.dark.json` is emitted, and (if both) triggers a parallel two-mode build below, so never assume it. Ask in one focused, grouped round; a second round only if an answer materially changes what else matters. Capture the answers (including `modes`) as a brand brief.
2. **Dispatch the plan-phase workers (parallel ‚Äî independent):**
   - `@design-system-wizard:token-architect` ‚Äî propose the **two-tier taxonomy**: a `primitives` collection (raw color/spacing/radius/type scales) and a `semantic` collection that aliases primitives, with semantic names **engineered to emit the exact identifiers the target component library hard-codes** (`--background`, `--foreground`, `--primary`, `--primary-foreground`, `--border`, `--ring`, `--radius`, ‚Ä¶), plus Light/Dark modes as the theming axis. **Ground every scale in a proven reference per the Craft & Measurement Standard** ‚Äî color in **Radix Colors** (12-step, light+dark) or shadcn defaults; spacing on the **4px grid** (Tailwind scale); a modular **type scale** (as text styles) with correct line-heights; radius + a 5-step elevation scale; **icons = Lucide**. Never hand-mix bland ramps. Every proposed semantic token is surfaced by name, with its grounding source cited.
   - `@design-system-wizard:component-planner` ‚Äî propose the base component inventory (each backed by a Radix/shadcn primitive, icons from **Lucide**) and the Component Contract doc structure each will ship with.
3. **Write the DS plan.** One written document covering: the brand brief; the proposed token taxonomy with **every semantic token named** and its grounding source; the component inventory; the **canonical Figma file structure** (`Cover ¬∑ Foundations ¬∑ Components ¬∑ Patterns`) and the foundation pages to render (Color, Typography, Spacing, Radius, Elevation, **Icons (Lucide)**, + conditional Motion); and the manifest that will be produced. Surface real choices (naming, font, token structure) as questions ‚Äî do not decide them on the human's behalf.

**STOP. Present the plan and WAIT for the human's explicit approval. Do not create, write, bind, or push anything until the human approves.**

## Execute phase (only after approval)

Run in order ‚Äî each step depends on the last. Figma access goes through `figc-operator` (the canonical operator) except the canvas-docs step, which `figma-doc-builder` runs itself under the same figc conventions; all code writes land on a git branch, never `main`.

1. `@design-system-wizard:figc-operator` ‚Äî build the `primitives` and `semantic` variable collections and the **modes the interview chose** (only the mode axis the user picked ‚Äî don't create a Dark mode if they chose Light only). **When both Light + Dark were chosen, build the two modes as a parallel fan-out**: dispatch figc-operator work per mode concurrently (each resolves every semantic role's value for its mode), then reconcile ‚Äî every semantic variable must carry a resolved value in *both* modes before proceeding. Bind semantic tokens as aliases of primitives (never hardcoded colors); build **typography as real Figma text styles**; import the **Lucide** icon set as SVGs into the Icons foundation; build the base components. **Everything is Figma auto-layout with token-bound gap/padding on the 4px grid ‚Äî never absolute-positioned, never overlapping** (per the Craft & Measurement Standard). After every canvas write, `figc shot`-verify and inspect.
2. `@design-system-wizard:token-writer` ‚Äî emit the DTCG token JSON (`primitives.json`, `typography.json`, and one `semantic.<mode>.json` **per chosen mode** ‚Äî both `semantic.light.json` + `semantic.dark.json` only if both modes were chosen; a single `semantic.<mode>.json` otherwise) with aliases preserved as references. Record the chosen modes in the manifest's `figma.modes`.
3. `@design-system-wizard:doc-writer` ‚Äî scaffold the Component Contract docs (code/web surface) per the Component Contract convention.
4. `@design-system-wizard:figma-doc-builder` ‚Äî render the **canvas documentation** into the canonical file structure (`Cover ¬∑ Foundations ¬∑ <one page per component> ¬∑ Patterns`): foundation pages (Color swatch grid bound to the actual variables, Typography specimens from the real text styles, Spacing, Radius, Elevation, **Icons (Lucide)**) and **one dedicated documentation page per component**, each a usage-first designed page with **use-case frames** (real instances + explanatory text showing how to use it), a variant/state matrix, anatomy, and Do/Don't. Optionally emit a companion **Markdown usage guide** per component for readability. **This layer is mandatory ‚Äî never skip it; every component gets its own page; within each page nothing overlaps and everything is auto-layout on the grid.**
5. **Craft-verification gate ‚Äî you (this command) own the loop.** Dispatch `@design-system-wizard:craft-reviewer` to `figc shot` + `figc bound` the foundation pages and every component frame and inspect against the **Craft Checklist** (no overlaps; all spacing on-grid/token-bound; radii from scale; type from scale; Lucide icons on-grid; auto-layout everywhere; contrast AA). craft-reviewer is a read-only inspector ‚Äî it returns a `pass`/`fail` report with per-node **work-orders**, it does not fix or re-shoot itself. On `fail`: dispatch `@design-system-wizard:figc-operator` to apply its work-orders, then **re-invoke craft-reviewer** to re-inspect. Repeat this dispatch‚Üífix‚Üíre-inspect loop until it returns `pass`. Do not report success until it does.
6. Write `ds-manifest.json` (brand brief + two-tier taxonomy + component inventory pointing at each Contract + parity map + canvas doc refs) and **validate it against `${CLAUDE_PLUGIN_ROOT}/schema/ds-manifest.schema.json`**.

**Exit criterion ‚Äî a solid Figma file.** `/setup`'s job is to leave the Figma file in the **solid state** the **Figma Readiness Gate** requires (variables resolved in every chosen mode ¬∑ typography as real text styles ¬∑ Foundations pages built ¬∑ core components built), plus a documentation page per component. Because Figma has priority and code follows, this solid state is what unblocks `/infra`. Before reporting done, confirm the Readiness Gate criteria are met.

Report what was built, the manifest path, the craft-review result, whether the **Figma Readiness Gate** now passes, and any follow-up (e.g. `/infra` to wire the design-to-code pipeline, now that the Figma file is solid).
